## 1) 변수 / 객체 / static 기본 개념

### 변수는 왜 만드나요?

→ **데이터를 저장하려고**

⇒ **메모리(Memory) 상의 공간에 저장하려고**

- **변수** : 데이터를 저장하는 공간(그릇)

### 객체는 왜 만드나요?

→ **실생활스럽게 관련 변수(속성)와 기능(메소드)을 묶어서 알아보기 편하게 하려고**

- 객체는 단순히 변수 1개만 두는 개념이 아니라,
    
    **관련된 속성과 기능을 하나로 묶기 위해** 만든다.
    

### 멤버변수로 저장할 게 없음

→ 객체를 만들 필요가 없는 경우가 많음

⇒ 보통은 이럴 때 **static 메소드 기반**으로 작성함

```java
public static void sum(int n1_1, int n2_1) {
    System.out.println(n1_1 + n2_1);
}
```

### 같은 맥락으로 Java의 내장 클래스 중 하나인 Math

이것 또한 객체를 만들지 않고 사용한 경우

```java
System.out.println(Math.PI);
System.out.println(Math.sqrt(10)); // static 기반의 메소드
```

포인트

- `Math.PI` 는 **static final 상수**
- `Math.sqrt()` 는 **static 메소드**
- 즉, **객체 생성 없이 클래스명으로 바로 사용 가능**

# 2) 계산기 예제

## 예제 코드

```java
import java.util.Scanner;

public class SMain1 {
    public static void main(String[] args) {
        Scanner s = new Scanner(System.in);
        Calc c = new Calc();

        System.out.println("첫번째 정수:");
        c.n1 = s.nextInt();

        System.out.println("두번째 정수:");
        c.n2 = s.nextInt();

        c.sum = c.n1 + c.n2;
        c.printSum();

        c.sum(10, 20);      // 호출은 가능하지만, static 메소드는 클래스명으로 부르는 것이 권장됨
        c.sum(10, 20, 30);  // static X, 인스턴스 메소드
        Calc.sum(30, 50);   // static O

        // Calc.sum(10, 20, 30); // static이 아니라서 불가능

        System.out.println(Math.PI);
        System.out.println(Math.sqrt(10)); // static 기반의 메소드
    }
}
```

```java
public class Calc {
    int n1;
    int n2;
    int sum;

    public void printSum() {
        System.out.println("첫번째 정수는 " + this.n1 + " 두번째 정수는 " + n2);
        System.out.println("합계는 " + sum + " 입니다.");
    }

    public static void sum(int n1_1, int n2_1) {
        System.out.println(n1_1 + n2_1);
    }

    public void sum(int n1_1, int n2_1, int n3) {
        System.out.println(n1_1 + n2_1 + n3);
    }
}
```

포인트

1. `c.sum(10, 20);`
    - **호출은 가능**
    - 하지만 `static` 메소드는 **객체명보다 클래스명으로 호출하는 것이 정석**
    - 즉, `Calc.sum(10, 20);` 가 더 올바른 스타일
2. `Calc.sum(30, 50); // static O => 메모리를 덜 쓰고 해결`
    - 이 표현은 **완전히 틀린 건 아니지만 정확하게는**
    - **객체를 만들 필요가 없어서 인스턴스 생성 비용이 들지 않는다** 가 더 맞음
    - 무조건 “항상 메모리를 덜 쓴다” 보다는
        
        **객체 상태가 필요 없을 때 효율적이다.**

# 3) static

## 핵심 개념

**static**

→ **클래스 소속**

⇒ 객체마다 따로 생기는 것이 아니라

**클래스 기준으로 1개만 만들어져서 공유됨**

즉,

- **static 멤버변수** : 객체들의 공통 속성
- **static 메소드** : 객체 없이도 호출 가능

---

## 예제

```java
class Number {
    static int num = 0; // static 멤버변수 : 객체들의 공통속성
    int num2 = 0;       // 인스턴스 멤버변수 : 객체마다 따로 존재
}

public class SMain2 {
    public static void main(String[] args) {
        Number number1 = new Number();
        Number number2 = new Number();

        number1.num++;
        number1.num2++;

        System.out.println(number1.num);  // 1
        System.out.println(number1.num2); // 1
        System.out.println(number2.num);  // 1  -> static 공유
        System.out.println(number2.num2); // 0  -> 객체별 개별
    }
}
```

포인트

- `System.out.println(number1.num);` → **1**
- `System.out.println(number1.num2);` → **1**
- `System.out.println(number2.num);` → **1**
- `System.out.println(number2.num2);` → **0**

### 이유

- `num` 은 static 이라서 **number1, number2가 함께 공유**
- `num2` 는 static 이 아니라서 **각 객체가 따로 가짐**

`static` 변수는 사실 아래처럼 **클래스명으로 접근하는 게 권장**

```java
Number.num++;
System.out.println(Number.num);
```

# 4) 생성자 (Constructor)

## 개념

생성자 (Constructor)

→ **객체가 만들어질 때 뭔가 작업을 해야 한다면 생성자를 만든다**

예:

- 멤버변수 초기화
- 객체 생성 시 필요한 값 받기
- 객체 생성과 동시에 준비 작업 수행

### 생성자의 특징

- **리턴타입이 아예 없음**
- **메소드명이 클래스명과 같음**

**정확히는 생성자는 메소드가 아니다.**

- 모양은 메소드와 비슷하지만
- **객체 생성 시 호출되는 특별한 문법 요소.**

---

## 생성자를 만들지 않으면

⇒ 자바 컴파일러가 **기본 생성자(default constructor)** 를 만들어줌

**아무 생성자도 직접 만들지 않았을 때만** 기본 생성자를 자동으로 만들어줌

---

## 기본 생성자를 건드리면…

⇒ 자바 컴파일러가 기본 생성자를 안 만들어줌

**생성자를 하나라도 직접 정의하면** 컴파일러는 기본 생성자를 자동으로 만들지 않음

그래서

- `new 클래스명()` 형태로 객체를 만들려고 했는데
- 기본 생성자가 없으면 에러가 날 수 있음

특히 프레임워크나 라이브러리처럼

**시스템이 자동으로 객체를 생성하는 경우** 기본 생성자가 필요한 일이 많음

---

## 기본 생성자 예시

```java
public class Nike {

    public Nike() {
        // 기본 생성자
    }
}
```

---

## this 예시

```java
public void test(String prod_name) {
    // 멤버변수명 / 파라미터명이 같다면...
    // this. 붙이면 무조건 멤버변수
    // 안붙이면 가장 가까이에 있는 것(지역변수, 파라미터)

    System.out.println(prod_name);       // 파라미터
    System.out.println(this.prod_name);  // 멤버변수
}
```

정리

- `this.prod_name` → 멤버변수
- `prod_name` → 가장 가까운 변수(여기서는 파라미터)

---

## 생성자 오버로딩 예시

```java
public class Nike {
    String prod_name;
    double prod_size;
    int prod_price;

    public Nike() {
    }

    public Nike(String prod_name, double prod_size, int prod_price) {
        super();
        this.prod_name = prod_name;
        this.prod_size = prod_size;
        this.prod_price = prod_price;
    }
}
```

### `super();`

→ Java의 최상위 클래스인 `Object`의 생성자를 호출

- 모든 클래스는 특별한 경우가 아니면 `Object`를 상속받음
- 생성자 첫 줄에 `super();` 가 들어가며,
- 부모 생성자를 호출함
- 안 써도 컴파일러가 자동으로 넣어주는 경우가 많음

# 6) 객체간의 관계

## 1) A has a B : 포함관계

예:

- 사람 has a 핸드폰
- 컴퓨터 has a 회사 정보
- 식당 has a 사장
- 메뉴 has a 식당

즉, **한 객체가 다른 객체를 멤버변수로 가짐**

---

## Computer / Company 예제

필기 원문에 아래 부분은 **타입명, 변수명, 출력용 데이터가 조금 섞여 있었음**

그래서 **오타와 충돌만 수정해서** 정리하면 이렇게 보는 게 맞음.

### 예시

- 삼성, 수원, 300명이 다니는 회사의 정보를 출력
- CPU i7-1234, RAM 32GB, HDD 500GB인 삼성 컴퓨터
- 컴퓨터 / 회사 정보를 출력

---

## 회사 클래스 예시

```java
public class Compa {
    String compa_name;
    String compa_loc;
    int compa_emp;

    public Compa(String compa_name, String compa_loc, int compa_emp) {
        super();
        this.compa_name = compa_name;
        this.compa_loc = compa_loc;
        this.compa_emp = compa_emp;
    }

    public void printCompaBIO() {
        System.out.println("회사명: " + compa_name);
        System.out.println("위치: " + compa_loc);
        System.out.println("직원수: " + compa_emp);
    }
}
```

## 컴퓨터 클래스

```java
public class ComputerCompa {
    String cpu;
    String ram;
    int hdd;
    Compa brand;

    public ComputerCompa(String cpu, String ram, int hdd, Compa brand) {
        super();
        this.cpu = cpu;
        this.ram = ram;
        this.hdd = hdd;
        this.brand = brand;
    }

    public void printCom() {
        System.out.println("CPU: " + cpu);
        System.out.println("RAM: " + ram + "GB");
        System.out.println("HDD: " + hdd + "GB");
        System.out.println("회사 명: " + brand.compa_name);
    }
}
```

## 실행 예시

```java
Compa cb = new Compa("삼성", "수원", 300);
cb.printCompaBIO();

ComputerCompa cc = new ComputerCompa("i7-1234", "32", 500, cb);
cc.printCom();
```

포인트

1. `Compa brand;` 인데 생성 예시는 `CompaBIO cb` 로 되어 있었음
    
    → **타입명을 하나로 통일해야 함**
    
2. `ComputerCompa cc = new ComputerCompa("CPU i7-1234", "RAM 32", 500, co);`
    - `co` 는 오타 → `cb`
    - `"CPU i7-1234"` 를 넘기면 출력이 `CPU: CPU i7-1234` 가 됨
    - `"RAM 32"` 를 넘기면 출력이 `RAM: RAM 32GB` 가 됨
        
        → 그래서 `"i7-1234"`, `"32"` 처럼 넣는 게 자연스러움

**접시의 가격(5000원)** 이므로

```java
this.price
```

가 맞음

# 8) A is a B : 상속관계

## 개념

```java
// A is a B : 상속관계
// 강아지는 동물이다.
```

### 예

- Dog is a Animal
- Cat is a Animal

즉, **하위 클래스가 상위 클래스의 속성과 기능을 물려받는 관계**

---

## Dog 예제

```java
public class Dog extends Animal {
    String inhabit;

    public Dog(String sort, int age, String inhabit) {
        super(sort, age);
        this.inhabit = inhabit;
    }

    @Override
    public void printAnimal() {
        super.printAnimal();
        System.out.println(inhabit);
    }
}
```

포인트

- **Dog 클래스가 Animal 클래스를 상속한다**
- 또는
- **Dog is a Animal**

---

## Annotation

`@xxx : Annotation`

⇒ 어떤 기능을 자동으로 연결해준다

이 중에서 중요한 건 **`@Override`**

### `@Override`

→ 상속받아온 메소드를 **재정의(다시 정의)** 하는 것

```java
@Override
public void printAnimal() {
    super.printAnimal();
    System.out.println(inhabit);
}
```

설명

- 부모 클래스의 메소드와
- **메소드명, 파라미터, 리턴타입이 맞아야** 오버라이딩 가능
- `super.printAnimal();`
    
    → 부모의 원래 기능을 먼저 실행하고,
    
    그 뒤에 자식 기능을 추가할 수 있음
    

---

## Overloading 과의 차이

### Overriding

- **상속 관계에서**
- 부모의 메소드를 자식이 **다시 정의**

### Overloading

- **같은 클래스 안에서**
- **메소드명은 같고**
- **파라미터의 개수나 타입이 달라야 함**

즉,

- `sum(int, int)`
- `sum(int, int, int)`

이건 **오버로딩**

---

## 상속이 필요한 이유

상속이 필요한 이유는 **코드 중복을 없애기 위해서**

코드 중복이 많아지면

- 개발 단계에서 피곤하고
- 유지보수 비용도 커짐

그래서 OOP에서는

**공통 속성과 기능을 부모 클래스에 두고**

**자식 클래스가 물려받는 구조**를 사용함

## Animal 클래스

```java
public class Animal {
    String sort;
    int age;

    public Animal(String sort, int age) {
        super();
        this.sort = sort;
        this.age = age;
    }

    public void printAnimal() {
        System.out.println("종: " + sort + "과");

    }
```

## Cat 클래스

```java
public class Cat extends Animal {

    public Cat(String sort, int age) {
        super(sort, age);
    }

    public static void main(String[] args) {
        Cat c = new Cat();
        c.printInfo();
    }
}
```

### 틀린 점 1

```java
Cat c = new Cat();
```

불가능

이유:

- `Cat(String sort, int age)` 생성자만 만들었기 때문에
- **기본 생성자 `Cat()` 가 없음**

### 틀린 점 2

```java
c.printInfo();
```

불가능

이유:

- `Cat` 클래스나 `Animal` 클래스에 `printInfo()` 가 없음
- 있는 메소드는 `printAnimal()`

# 10) 핵심 요약

## 변수

- 데이터 저장용 공간

## 객체

- 관련된 변수(속성)와 메소드(기능)를 묶은 것

## 멤버변수

- 객체 안에 들어있는 변수
- 객체의 상태를 저장

## static

- 객체 소속이 아니라 **클래스 소속**
- 객체 없이 사용 가능
- static 변수는 모든 객체가 공유
- static 메소드는 객체 정보 없이 동작할 때 적합

## this

- 현재 객체 자신
- 멤버변수와 파라미터 이름이 같을 때 구분할 때 사용

## 생성자

- 객체 생성 시 호출
- 리턴타입 없음
- 클래스명과 이름이 같음
- **메소드는 아님**
- 생성자를 하나도 안 만들면 기본 생성자 자동 제공
- 하나라도 만들면 기본 생성자는 자동 생성 안 됨

## 오버로딩

- 같은 이름의 메소드
- 파라미터 개수/타입이 다름

## 포함관계 (has-a)

- 객체 안에 다른 객체를 멤버변수로 포함
- 예: 사람 has a 핸드폰

## 상속관계 (is-a)

- 자식 클래스가 부모 클래스의 속성과 기능을 물려받음
- 예: Dog is a Animal

## 오버라이딩

- 상속받은 메소드를 자식 클래스에서 다시 정의
- `@Override` 사용

## super

- 부모 생성자 호출: `super()`
- 부모 메소드 호출: `super.메소드명()`

# 요약 정리

## 1. 변수 / 객체

- **변수 = 데이터 저장 공간**
- **객체 = 속성 + 기능을 묶어놓은 것**

## 2. static

- **static = 클래스 것**
- 객체 없이 `클래스명.메소드명()` 으로 사용
- static 변수는 **공유**
- 인스턴스 변수는 **객체마다 따로**

예:

- `Math.PI`
- `Math.sqrt(10)`

## 3. 생성자

- 객체 만들 때 실행
- **리턴타입 없음**
- **이름이 클래스명과 같음**
- **메소드 아님**
- 생성자 안 만들면 기본 생성자 자동 생성
- 단, **직접 생성자 하나라도 만들면 기본 생성자 자동 생성 안 됨**

## 4. this / super

- `this` = 내 객체
- `super` = 부모 객체 쪽

## 5. 오버로딩 / 오버라이딩

- **오버로딩** = 같은 이름, 파라미터 다름
- **오버라이딩** = 상속받은 메소드 재정의

## 6. 객체 관계

- **has-a** = 포함관계
    
    예: 사람 has a 핸드폰
    
- **is-a** = 상속관계
    
    예: 강아지 is a 동물
    

## 7. 추가

- static 메소드를 객체로 호출할 수는 있지만 **클래스명으로 호출하는 게 정석**
- 생성자 정의했으면 기본 생성자 자동 생성 안 됨
- `@Override` 는 부모 메소드와 **형태가 정확히 같아야** 함
- static 변수는 객체별이 아니라 **공유**
- 인스턴스 변수는 **객체별 따로**
